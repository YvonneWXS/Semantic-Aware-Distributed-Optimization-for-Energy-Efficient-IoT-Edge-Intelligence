# SA‑IPPO 算法修改记录

本文档详细记录了相对于原始 SA‑IPPO 代码的所有修改内容、修改原因及其影响。

## 一、动作空间扩展

### 1.1 修改文件
- `envs/env_discrete.py`
- `envs/env_core.py`

### 1.2 修改内容
**原始动作空间**（三维）：
- `[offload_decision, semantic_factor, resource_allocation]`
- 其中 `resource_allocation` 实际对应 MEC 资源分配比例，但原代码中误标为“资源分配”，而传输功率被单独编码为第三个维度。

**新动作空间**（四维）：
- `[offload_decision, semantic_factor, resource_allocation, bandwidth_weight]`
- 删除原有的传输功率维度，新增带宽权重维度。

**具体改动**：
1. `env_discrete.py` 中 `action_space_params` 从 `[[0,1], [0,99], [0,94]]` 改为 `[[0,1], [0,7], [0,9], [0,3]]`。
   - 卸载决策：0/1（2 个值）
   - 语义因子：8 个离散值（对应 0.3~1.0，步长 0.1）
   - MEC 资源：10 个离散值（对应 0.1~1.0，步长 0.1）
   - 带宽权重：4 个离散值（0,1,2,3）
2. `env_core.py` 中 `self.action_dim` 从 3 改为 4。
3. 动作解析逻辑从原来的三个部分（卸载 2、资源 10、功率 50）改为四个部分（卸载 2、语义 8、资源 10、带宽权重 4），总 one‑hot 维度从 62 变为 24。

### 1.3 修改原因
1. **遵循论文要求**：论文中明确将带宽分配作为优化变量，而传输功率在后续分析中被简化或固定。
2. **资源解耦**：将带宽分配从固定的均分模式改为由智能体表达需求强度，符合“通信‑计算‑语义”联合优化的核心思想。
3. **降低学习难度**：离散权重比连续功率值更易于学习，且能与语义因子、MEC 资源形成统一的离散动作表示。

### 1.4 影响
- 智能体需同时学习带宽需求与语义压缩、卸载决策之间的平衡。
- 网络输出层维度变化，需重新训练。
- 传输功率被固定为 0.1 W（可在 `env_core.py` 中修改），简化了功率控制问题。

## 二、DW‑DNA 带宽分配机制

### 2.1 修改文件
- `envs/env_core.py` 的 `step` 函数

### 2.2 修改内容
**原始带宽分配**：
```python
if offload_num > 0:
    W = self.transmission_bandwidth / offload_num  # 静态均分
```

**新带宽分配（DW‑DNA）**：
1. 收集所有智能体的带宽权重 `w_i`。
2. 计算总权重 `sum_w`。
3. 若 `sum_w > 0`，对每个智能体计算理想分配比例 `β_i = w_i / sum_w`。
4. 理想带宽 `B_ideal = β_i * total_bandwidth`。
5. 向下取整到最近的 RB 整数倍：`num_RBs = floor(B_ideal / RB_bandwidth)`，实际带宽 `B_i = num_RBs * RB_bandwidth`。
6. 若 `sum_w == 0`，所有 `B_i = 0`。

其中 `RB_bandwidth = 180 kHz`（5G NR 标准）。

### 2.3 修改原因
1. **物理真实性**：5G/6G 中带宽分配以 RB 为单位，向下取整模拟了实际系统的离散性。
2. **避免死锁与浪费**：归一化机制自动保证总带宽不超限，且权重为 0 的用户不占带宽，避免了静态均分导致的资源浪费。
3. **算法友好性**：奖励函数对带宽权重的变化是平滑的（带宽随权重增加而单调增加），有利于策略梯度更新。

### 2.4 影响
- 带宽分配从“均分”变为“按需竞争”，智能体可通过调整权重表达不同的传输需求。
- 引入 RB 量化后，实际分配的带宽可能略低于理想值，剩余带宽被视为系统保护间隔，这更符合实际通信场景。
- 若所有权重均为 0，则所有用户带宽为 0，卸载任务将无法传输（时延、能耗为无穷大），智能体必须学习避免该情况。

## 三、时延与能耗公式更新

### 3.1 修改文件
- `envs/env_core.py` 的 `step` 函数中卸载分支的计算部分

### 3.2 修改内容
**原始公式**：
```python
uplink_rate = W * math.log2(1 + transmission_power * channel_gain / (W * noise_power))
```

**新公式**：
```python
if W_i > 0:
    uplink_rate = W_i * math.log2(1 + self.transmission_power * self.channel_gain[i] / (W_i * self.noise_power))
else:
    uplink_rate = 0.0
```

其中 `W_i` 为通过 DW‑DNA 分配给用户 i 的实际带宽。

### 3.3 修改原因
- 带宽分配从全局均分变为个体差异化，每个用户的上行速率必须使用其独享的带宽计算。
- 若分配带宽为 0，理论上传输无法进行，此处设为 0 速率并会在后续产生极大惩罚，促使智能体学习合理的权重选择。

### 3.4 影响
- 时延、能耗与带宽权重形成非线性关系：权重增加 → 带宽增加 → 速率提高 → 传输时延降低 → 总时延可能降低。
- 这种耦合关系正是 Reviewer 期望看到的“通信‑语义”联合优化效果。

## 四、批量实验支持

### 4.1 新增文件
- `config_batch.py`
- `main.py`

### 4.2 修改内容
1. `config_batch.py` 定义了五组参数列表：
   - `DataSize_list`: 任务数据大小
   - `num_UEs_list`: 用户设备数量
   - `bandwidth_list`: 系统总带宽
   - `mec_capacity_list`: MEC 计算能力
   - `min_semantic_factor_list`: 语义因子下限
2. `main.py` 提供了参数遍历框架，可自动生成实验名称并调用训练脚本。

### 4.3 修改原因
- 系统性能受多种参数影响，批量实验能全面评估算法在不同场景下的鲁棒性。
- 为后续的消融实验、对比实验提供基础设施。

### 4.4 影响
- 用户可通过修改 `config_batch.py` 快速设置多组实验。
- `main.py` 目前为框架代码，实际运行时需根据实验设计调整环境参数的传递方式（建议直接修改 `env_core.py` 中的对应变量）。

## 五、保持不变的部分

为确保与原始论文的一致性，以下部分未作改动：

1. **语义提取计算模型**：公式 `E_sem = κ * α * D^r * (β^(-k) - 1) * f^2` 完全保留。
2. **本地执行与 MEC 执行公式**：时延、能耗计算公式除带宽部分外均未变动。
3. **IPPO 网络结构**：算法部分的 `rMAPPOPolicy`、`r_actor_critic`、`r_mappo` 等未修改。
4. **PopArt 与 PPO 裁剪机制**：正常使用。
5. **奖励函数结构**：仍然为“全局能耗效率 + 时延惩罚”的形式，权重 `w1, w2` 可调。

## 六、已知限制与后续改进建议

1. **固定传输功率**：目前传输功率固定为 0.1 W，若需研究功率控制，可将其恢复为动作维度或作为环境参数随机变化。
2. **带宽量化损失**：RB 向下取整会导致少量带宽浪费，可在后续工作中加入剩余带宽的再分配策略。
3. **无效动作组合**：带宽权重为 0 但卸载决策为 1 的组合会导致传输失败，目前仅通过极大惩罚让智能体自学，可考虑在动作选择时增加约束。
4. **参数传递**：批量实验时环境参数（如带宽、MEC 能力）需手动修改 `env_core.py`，未来可通过命令行参数或配置文件动态传入。

## 七、总结

本次修改严格遵循论文要求，完成了动作空间扩展、DW‑DNA 带宽分配、公式适配三项核心任务，并提供了批量实验基础设施。修改后的代码保持了原始 SA‑IPPO 的算法框架与奖励结构，同时引入了更符合实际通信系统的带宽分配机制，为“语义‑通信‑计算”联合优化的深入研究奠定了代码基础。
